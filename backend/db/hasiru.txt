Great questions. Let me explain both.

Diet Conversion Factors are specifically for Pol Sambola and Soup recipes only. They solve this problem: a 2-year-old child doesn't eat the same portion as an adult, so when calculating recipe ingredients, children's counts are scaled down.

Here's how it works with a real example from the PDF:

If 15 children (S1) are in the hospital and Pol Sambola is ordered, the system doesn't count them as 15 full portions. Instead it converts:
Diet Type	Patient Count	× Factor	= Actual Portions
Normal	47	× 1.0	47
Diabetic	24	× 1.0	24
S1 (6-12y)	15	× 0.72	10.8
S2 (2-6y)	5	× 0.72	3.6
S3 (1-2y)	2	× 0.48	0.96
Staff	10	× 1.0	10
Total			96.36 actual portions

Then each Pol Sambola ingredient is multiplied by 96.36 (not by the raw 103 headcount). So Coconut at 0.1 seeds per patient = 9.6 seeds needed.

These factors only apply to recipes (Pol Sambola, Soup, Kanda) — not to regular items like rice or vegetables. Regular items use the norm weight matrix where each diet type already has its own gram value built in.

So this section in the UI is correct and important — keep it.

For the decimal point issue on norm weights, you're right — the norm weight editor needs to allow decimals because several values require them (e.g., Egg for S3 = 0.5, Coconut per patient = 0.1, dried fish norms, bread = 0.25 loaves). Here's the fix for your norm weight input fields:

Change the number inputs from:
jsx

<Input type="number" defaultValue={value} />

To:
jsx

<Input type="number" step="0.01" min="0" defaultValue={value} />

The step="0.01" allows two decimal places. If you want even finer control (for values like 0.125), use step="0.001".

    await client.query('BEGIN');
    
    const query = 
      INSERT INTO norm_weights (item_id, diet_type_id, meal, weight)
      VALUES ($1, $2, $3, $4)
      ON CONFLICT (item_id, diet_type_id, meal) DO UPDATE SET
        weight = EXCLUDED.weight,
        updated_at = CURRENT_TIMESTAMP
      RETURNING *;
    ;
    const updatedRows = [];
    for (const data of weightsData) {
      // Skip if dietTypeId is missing somehow
      if (!data.dietTypeId) continue; 
      
      const values = [
        itemId,
        data.dietTypeId,
        data.meal,
        data.weight || 0
      ];
      const res = await client.query(query, values);
      updatedRows.push(mapWeightRow(res.rows[0]));
    }
    await client.query('COMMIT');
    return updatedRows;
  } catch (error) {
    await client.query('ROLLBACK');
    throw error;
  } finally {
    client.release();
  }

-- Daily calculation snapshots (one run per date)
CREATE TABLE IF NOT EXISTS calculation_runs (
    id SERIAL PRIMARY KEY,
    calc_date DATE NOT NULL UNIQUE,
    patient_cycle VARCHAR(100) NOT NULL,
    staff_cycle VARCHAR(100) NOT NULL,
 
    -- Aggregated patient totals (snapshot)
    patient_totals JSONB NOT NULL DEFAULT '{}'::jsonb,
    -- e.g. {"NOR": 47, "DM": 24, "S1": 15, "S2": 5, "S3": 2, ...}
 
    staff_breakfast INTEGER NOT NULL DEFAULT 0,
    staff_lunch INTEGER NOT NULL DEFAULT 0,
    staff_dinner INTEGER NOT NULL DEFAULT 0,
 
    -- Special request totals (hospital-wide)
    soup_count INTEGER NOT NULL DEFAULT 0,
    kanda_count INTEGER NOT NULL DEFAULT 0,
    pol_sambola_count INTEGER NOT NULL DEFAULT 0,
 
    -- Extra items aggregated (raw sums across wards)
    extras_totals JSONB NOT NULL DEFAULT '{}'::jsonb,
    custom_extras_totals JSONB NOT NULL DEFAULT '[]'::jsonb,
 
    status VARCHAR(30) NOT NULL DEFAULT 'calculated',
    -- calculated → draft_po → pending_approval → approved → invoiced
 
    calculated_by INTEGER REFERENCES users(id),
    calculated_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
 
    created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);
 
-- Individual calculation line items (one per item × meal)
CREATE TABLE IF NOT EXISTS calculation_line_items (
    id SERIAL PRIMARY KEY,
    calc_run_id INTEGER NOT NULL REFERENCES calculation_runs(id) ON DELETE CASCADE,
    item_id INTEGER NOT NULL REFERENCES items(id),
    meal VARCHAR(20) NOT NULL, -- 'breakfast', 'lunch', 'dinner', 'extra', 'recipe'
 
    -- Per-diet-type breakdown (grams in base unit)

    base_unit VARCHAR(10) NOT NULL DEFAULT 'g',       -- 'g', 'ml', 'pcs'
 
    display_value NUMERIC(14,4) NOT NULL DEFAULT 0,   -- Converted to display unit
    display_unit VARCHAR(30) NOT NULL DEFAULT 'Kg',    -- Item's preferred unit
 
    created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
 
    CONSTRAINT uq_calc_item_meal UNIQUE (calc_run_id, item_id, meal)
);
